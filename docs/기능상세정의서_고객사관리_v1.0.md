# 고객사 정보 수집·관리 — 기능상세정의서 v1.0

> 작성일: 2026-07-12 · 대상 시스템: 나라장터 입찰정보시스템(확장) · 상태: 정의(구현 전)
> 회사: **(주)가덕씨엔에스** — 정보시스템감리·컨설팅·PMO·보안 전문
> 데이터 출처: 회사 홈페이지 고객사 정보 게시판 `http://www.gadukcns.com/41` (총 39곳, 라이브 수집 완료)

이 문서는 회사 **고객사(발주·수요기관) 정보를 수집·관리**하고, 이를 **나라장터 입찰공고와 연계**하여 우선순위·재수주·영업관리에 활용하는 기능을 상세화한다. **현재 프로젝트(Supabase + Next.js)에 즉시 적용 가능**하도록 기존 스키마·RLS·함수·컴포넌트 패턴을 재사용한다.

---

## 1. 개요 / 목적

- 회사 고객사 39곳(중앙부처·지자체·공공·의료·교육·금융)의 정보를 **DB로 관리**한다.
- 나라장터 공고의 **발주기관/수요기관이 고객사와 매칭**되면 **우선정렬·배경 강조·재수주 알림**으로 영업 기회를 부각한다.
- 고객사 목록은 **회사 홈페이지에서 자동 수집** + **수동 관리(CRUD)** 를 병행한다.

## 2. 배경 / 현황 (라이브 수집 결과)

- 고객사 **총 39곳**, 카테고리 6종(중앙정부부처·지방자치단체·공공기관·의료기관·교육기관·금융기관·기타).
- **의료기관(병원) 17곳 + 보건·복지 중앙부처 다수** → 회사의 **보건·의료 분야 정보시스템 감리·컨설팅** 강세 확인.
- 앞선 "우선순위 발주기관" 검토(`기능검토_발주기관_우선순위.md`)의 **실데이터 원천**이 바로 이 고객사 목록이다.

## 3. 데이터 모델 (기존 스키마에 신규 추가)

### 3.1 `clients` — 고객사 마스터
```sql
create table if not exists clients (
  client_id   bigserial primary key,
  name        text not null unique,              -- 정식 기관명 (예: '질병관리청')
  category    text not null check (category in
              ('중앙정부부처','지방자치단체','공공기관','의료기관','교육기관','금융기관','기타')),
  aliases     text[] default '{}',               -- 매칭용 별칭/약칭 (예: ['질병청'])
  sector      text,                              -- 분야 (예: '보건·의료')
  region      text,                              -- 지역
  is_priority boolean default true,              -- 우선 고객사 여부(정렬·강조 대상)
  weight      int default 10,                    -- org 룰 가중치(연동)
  contact_name text,
  contact_hash text,                             -- 담당자 연락처 HMAC(개인정보)
  memo        text,
  source_url  text,                              -- 수집 출처
  first_year  int,                               -- 최초 거래연도(선택)
  status      text default 'active' check (status in ('active','inactive')),
  created_at  timestamptz default now(),
  updated_at  timestamptz default now()
);
create index if not exists idx_clients_category on clients(category);
create index if not exists idx_clients_priority on clients(is_priority);
```

### 3.2 매칭(파생) — 공고 ↔ 고객사
- 별도 저장 없이 조회 시 계산하거나, 성능 위해 `bids.client_id`(nullable FK) 캐시 컬럼을 선택 추가.
- 매칭 규칙: `clients.name` 또는 `aliases[*]` 가 `bids.order_org` **또는** `bids.demand_org`에 포함(부분일치, 소문자·공백 정규화).
  - **수요기관(demand_org)도 매칭** — 조달청 대행 공고(발주=조달청, 수요=질병관리청)를 실수요기관 기준으로 잡기 위함.

### 3.3 RLS (기존 정책 패턴 준수)
```sql
alter table clients enable row level security;
create policy clients_read  on clients for select using (app_current_role() is not null);      -- active 전원 R
create policy clients_write on clients for all    using (app_current_role() in ('pm','admin')) -- pm/admin W
                                       with check (app_current_role() in ('pm','admin'));
```

## 4. 기능 요구사항 (FR-16 ~ FR-20 — 기존 FR-15 다음 번호)

### FR-16 고객사 CRUD (수동 관리)
- **개요**: 고객사 등록/수정/삭제, 카테고리·별칭·우선순위·가중치 관리.
- **처리**: `clients` upsert(name 유니크). 별칭 배열 편집. is_priority 토글.
- **권한**: pm/admin(RLS). **관련테이블**: clients.
- **예외**: 중복 name → 병합 안내.

### FR-17 고객사 자동 수집
- **개요**: 회사 홈페이지 고객사 게시판을 파싱해 `clients` 자동 upsert.
- **소스**: `http://www.gadukcns.com/41` (게시판, 페이지네이션 4p, 카테고리 6종). ※ HTTPS 자체서명 → **HTTP 사용**, 인코딩 UTF-8.
- **처리**: Node(undici) fetch → HTML 파싱(기관명+카테고리 추출) → name 기준 upsert(신규만 추가, 기존 유지). 수동 실행 또는 주 1회 배치.
- **주의**: 자사 사이트이므로 수집 허용. robots/이용약관 준수. 기관명 외 개인정보 미수집.
- **관련테이블**: clients. **산출물**: `scripts/collect_clients.mjs`.

### FR-18 공고-고객사 매칭 & 우선순위/강조
- **개요**: 수집된 공고(bids)의 발주/수요기관을 고객사와 매칭 → **우선정렬 + 배경 강조 + 고객사 배지**.
- **처리**:
  1. `is_priority` 고객사를 **`rules(type='org')`로 동기화**(name·별칭 → org 룰, weight 반영) → 재스코어링 시 agency_bonus 부여.
  2. S-04/S-05 목록: `.order('score', desc)`(우선기관 상단) + 카드 배경 틴트(`--color-priority-bg`) + `⭐ 고객사` 배지.
  3. 상세(S-06): "고객사 공고" 표식 + 고객사 카테고리 표시.
- **권한**: active 전원 R. **관련테이블**: clients, bids, rules.

### FR-19 고객사 현황/통계
- **개요**: 고객사별 공고 수·재수주 현황·관심목록 연계 지표.
- **처리**: 카테고리별/고객사별 집계, 최근 공고, watchlist 연계. S-10(통계) 또는 S-14 내 표시.
- **관련테이블**: clients, bids, watchlist.

### FR-20 우선 고객사 ↔ 스코어링 룰 동기화
- **개요**: `clients.is_priority=true` 변경 시 대응 `rules(type='org')` 자동 생성/갱신(가중치 = clients.weight).
- **처리**: DB 함수 `sync_client_org_rules()`(SECURITY DEFINER, admin) 또는 저장 트리거. 이후 재스코어링으로 반영.
- **관련테이블**: clients, rules.

## 6. 프로젝트 적용성 (요구: "현재 프로젝트에 적용가능")

| 재사용 자산 | 적용 |
|-------------|------|
| 스키마·RLS 패턴 | `clients` 1개 테이블 + 기존 RLS 정책 스타일 그대로 |
| DB 함수 패턴 | `rescore_bids`/`cleanup_bids`와 동일한 SECURITY DEFINER + admin 가드 (`sync_client_org_rules`) |
| org 룰 메커니즘 | **이미 존재**(agency_bonus) — 고객사 우선순위가 곧바로 점수·정렬에 연동 |
| 프론트 컴포넌트 | `BidCard`(배지·배경), `PageHeader`, react-query, RoleGuard 재사용 |
| 정렬/강조 | 앞 검토(`기능검토_발주기관_우선순위.md`)와 동일 구현 |
| 시드 데이터 | **고객사 39곳 이미 수집 완료** → 즉시 clients 시드 가능 |

- **신규 DDL은 `clients` 테이블 + 함수 1개**뿐. 나머지는 프론트 수정.
- 적용 절차: ① `clients` 스키마 SQL 적용(SQL Editor) → ② 39곳 시드 → ③ org 룰 동기화 + 재스코어링 → ④ S-14 + S-04 연계 UI.

## 7. 수집·매칭 상세 (FR-17/18 알고리즘)

- 정규화: 소문자화 + 공백 압축(기존 scoreBid `norm`과 동일).
- 매칭: `norm(order_org+demand_org).includes(norm(client.name))` 또는 별칭. 부분매칭 오탐 방지 위해 **정식명칭** 사용.
- 우선정렬: org 룰 가중 → 점수 상단. 순수 "고객사 먼저"가 필요하면 고객사 여부를 1순위 정렬키로.

## 8. 보안 / 개인정보

- 고객사명·카테고리: 공개 성격(회사 홈페이지 게시). 담당자 연락처는 **contact_hash(HMAC)** 로만 저장(FR-01 개인정보 원칙 준수).
- 자동 수집은 자사 사이트 대상. 외부 배포/재게시 안 함.

## 9. 테스트 시나리오

1. **정상**: FR-17 수집 → clients 39곳 → is_priority 동기화(org 룰) → 재스코어링 → S-04에서 질병관리청·삼성서울병원 등 공고 상단·배경강조.
2. **매칭(수요기관)**: 발주=조달청, 수요=질병관리청 공고 → 고객사 매칭·강조 확인.
3. **CRUD**: 신규 고객사 추가/별칭 등록 → 매칭 반영.
4. **권한**: 일반 사용자 clients write 차단(RLS).
